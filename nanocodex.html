一个刻意保持精简、但链路完整的 Codex 风格编码 Agent。它不是聊天壳，而是把模型工具调用、文件修改、命令执行、沙箱审批、会话记忆、MCP 扩展、上下文压缩和成本统计串成一条可运行的本地工程链路。

01项目定位

nanocodex 的目标不是复刻一个庞大的商业 IDE Agent，而是把 coding agent 最关键的工程机制拆开、写清楚、测扎实。

模型负责理解任务、规划下一步和提出工具调用；程序负责把工具调用变成真实的文件读写、patch、shell、计划更新、记忆写入和 MCP 调用。所有越权动作都会先经过沙箱策略与审批状态机，所有会话都会落到本地 JSONL 和快照里，方便回放、恢复和排查。

它更像一个“可研究、可调试、可替换后端”的 agent harness：你可以接 DeepSeek 托管 API，也可以把 base_url 指到本地 vLLM、llama-server、LM Studio 或其它 OpenAI 兼容服务，用同一套工具协议验证模型的 coding 能力。

它解决什么问题

把“模型会写代码”推进到“模型能在受控环境里持续读文件、改代码、跑命令、看结果、再修正”，并且每一步都能审计。

它为什么适合本地模型

后端只要求 OpenAI 兼容 /v1/chat/completions，所以可以在同一套 agent 工具链下比较 DeepSeek、Qwen、本地量化模型或企业内网模型。

它为什么适合讲工程能力

重点不在“套 API”，而在工具协议、权限边界、状态持久化、上下文压缩、成本统计、失败可恢复和可测试性。

02设计思想

nanocodex 不是要做「功能最全的 Agent」，而是用最少的概念把一个编码 Agent 的关键链路跑通、且每一步都可解释、可测试、可信任。整套代码贯穿四条原则。

核心循环刻意精简

一个回合就是「调用模型 → 执行工具 → 把结果喂回 → 继续」直到任务完成。Agent 不知道控制台、不知道 GUI，渲染靠 hook 注入，循环本身只关心推进任务。

纯函数 + 可注入 I/O

schedule、memory、skills、compaction、pricing、marketplace 的核心逻辑都是对数据的纯函数，时间、网络、文件都从外部注入。结果是整套 420 个测试全离线、不碰真实 API 与网络。

对局限性诚实

价格表是硬编码快照、带 as-of 日期，未知模型返回「成本未知」而不是错数；Windows 沙箱明说是策略级、非内核隔离；视觉模型支持标注为未验证。不假装能力。

无隐藏状态

会话是可读的 JSONL、记忆是用户自己的 markdown 文件、定时任务是一份 JSON。没有黑盒数据库，用户能直接看到、编辑、删除 Agent 依赖的一切。

03核心：一个回合的循环

AgentLoop.run_turn() 驱动单个用户回合直到完成。一个回合内可能发生多轮模型调用（每次工具使用算一轮），token 用量逐轮累计、折算成本。

用户输入 → 调用模型 → 解析工具调用 → 沙箱/审批闸门 → 执行工具 ↺ 回合完成 
# agent/loop.py — 回合产物
class TurnResult:
    final_text: str
    iterations: int          # 本回合循环了几轮
    stop_reason: str
    tools_used: list[str]
    usage: dict[str, int]    # 真实 token 累计 → 折算成本
1. 组装上下文读取 system prompt、AGENTS.md、记忆、可用 skills、MCP 工具 schema 和会话历史。
2. 调用模型走 DeepSeek / OpenAI 兼容 provider，支持流式输出、tool_calls 和 token usage。
3. 工具派发ToolRegistry 按名称路由到 shell、apply_patch、read_file、remember 或 MCP 工具。
4. 权限判断沙箱策略判断可读、可写、可联网，审批策略决定 ASK / AUTO_DENY / AUTO_APPROVE。
5. 结果回填工具结果以 tool message 回到模型上下文，模型继续决定下一步。
6. 持久化最终答案、工具轨迹、token 用量和会话快照写入本地，GUI 可回放。

04核心能力

能力做法
模型后端DeepSeek / 任意 OpenAI 兼容 /v1/chat/completions，支持流式；改 base_url 即可指向本地模型（vLLM / llama-server / LM Studio）。
工具集shell · apply_patch · update_plan · read_file · web_search · manage_schedule · manage_skills · remember，统一经 ToolRegistry 派发。
沙箱策略三种模式 read-only / workspace-write / danger-full-access，决定可写根目录与网络；策略与执行分离。
审批状态机四档 untrusted / on-failure / on-request / never，越权操作（含联网搜索）走 ASK / AUTO_DENY / AUTO_APPROVE。
上下文压缩deterministic（零成本规则摘要，默认）或 summarizer（消耗 token，opt-in）；保留 system + 近期尾部，且尾部必须从 user 消息开始。
自适应推理档按用户输入选 max / low / high；关键词表覆盖中日英（"报错""调试""デバッグ" 都能命中）。
成本统计cache-aware：命中 token 比未命中便宜约 120×，有拆分时分别计费，无拆分时按未命中保守计，从不低估。
会话历史JSONL 持久化；按 session_id 存全量快照，可在 GUI 左栏点选回放真实对话；支持 --resume。
记忆 / AGENTS.md / Skillsmemory 记「谁/什么」（用户偏好），skills 记「怎么做」，AGENTS.md 是项目级指令；skills 只注入名称+描述，正文按需读取。
MCP + 市场来自 ~/.nanocodex/mcp.toml，以 mcp__<server>__<tool> 暴露；运行在沙箱外，内置 catalog + 远程 catalog（按不可信数据处理）。
GUI 特性prompt 增强（✨）、# 快速记忆、定时任务、A/B worktree 对比（两套配置同 prompt 各跑一遍，对比 diff/成本/延迟，采纳其一）。

05工程取舍

  • 核心循环小：AgentLoop 只关心“模型、工具、结果、继续”，GUI、CLI、调度器都通过边界层装配，避免把展示逻辑塞进 agent 内核。
  • 工具协议显式：每个工具有 schema、参数校验和统一返回格式，模型不会直接碰文件系统或系统命令，必须通过工具层。
  • 安全不吹大：Windows 侧是策略级沙箱，不宣称内核隔离；MCP 明确运行在沙箱外，远程 marketplace catalog 按不可信数据处理。
  • 上下文压缩可控：默认 deterministic compaction，不额外花 token；需要更自然摘要时才 opt-in 模型 summarizer，避免长会话成本失控。
  • 成本统计保守：能拿到 cache hit / miss 就分别计费，拿不到就按未命中保守估算，宁可偏高，不给用户一个虚假的低成本。
  • 测试优先：provider、时间、文件、网络都可注入 mock，420 个测试离线跑通，保证 agent 行为可回归，而不是只靠手工试用。
一句话项目价值：nanocodex 展示的是 AI Agent 工程化能力：不是让模型“说会写代码”，而是让模型在可审计、可恢复、可限权、可测量成本的环境里完成真实编码任务。

07快速开始

# 安装（含开发依赖）
python -m pip install -e ".[dev]"

# CLI
nanocodex --cd .
nanocodex "add a --json flag to the CLI"

# GUI
nanocodex-gui --cd .

# 全套离线测试（无需真实 key / 网络）
python -m pytest -q
安全：真实 API key 永远留在仓库外（环境变量或 ~/.nanocodex/config.toml）。Windows 沙箱是策略级而非内核隔离；MCP 工具在沙箱外执行子进程，只启用你信任的服务。
